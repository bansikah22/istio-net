# Understanding the Kubernetes Gateway API

This document provides a beginner-friendly guide to the Kubernetes Gateway API, explaining what it is, why it's useful, and how to configure it.
## What is the Gateway API?

The Kubernetes Gateway API is the official, next-generation standard for managing network ingress in Kubernetes. It is designed to replace the older Ingress resource with a more expressive, powerful, and role-oriented model.

Think of it as a universal language for describing how to route traffic from outside your cluster to the services running inside it. Implementations like Istio, Contour, and various cloud-native load balancers all understand this "language."
## Why is it Better than Ingress?

The Gateway API improves upon the Ingress API in several key ways:

-   **Role-Oriented**: It separates the responsibilities of different user roles. A Cluster Administrator can manage the `Gateway` (the infrastructure), while Application Developers can manage their own `HTTPRoute` resources (the application routing) without interfering with each other.
-   **Expressive**: It provides richer features for traffic management, such as header-based matching, traffic splitting for canary releases, and more, all within the standard API.
-   **Extensible**: It's designed to be easily extended, allowing different implementations (like Istio) to add their own custom functionality in a standardized way.
-   **Portable**: Because it's an official Kubernetes standard, your routing configurations are more portable across different environments and ingress controllers.

## Core Concepts: Gateway and HTTPRoute

The two main resources you will work with are:

-   **`Gateway`**: This resource defines where and how a load balancer listens for traffic. It's typically managed by a cluster administrator. A `Gateway` specifies the `gatewayClassName` (which implementation to use, e.g., `istio`), the ports to open, and which Routes are allowed to attach to it.

-   **`HTTPRoute`**: This resource defines how HTTP requests are routed from a `Gateway` to a backend service. It's typically managed by an application developer. An `HTTPRoute` specifies which `Gateway` it attaches to, what kind of traffic to match (e.g., requests for `/`), and which service to forward that traffic to.

## How to Configure the Gateway API

To use the Gateway API, you need to perform two main steps:
### 1. Install the CRDs

The Gateway API resources (`Gateway`, `HTTPRoute`, etc.) are not installed in a Kubernetes cluster by default. You must first install their definitions, known as Custom Resource Definitions (CRDs).

Why do we do this? This command teaches your Kubernetes cluster about the Gateway API. Without it, the cluster wouldn't recognize `Gateway` or `HTTPRoute` resources, and `kubectl apply` would fail.

```sh
# This command applies the standard set of Gateway API CRDs to your cluster.
# You only need to do this once per cluster.
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.0.0/standard-install.yaml
```

### 2. Create the Gateway and HTTPRoute Resources

Once the CRDs are installed, you can create the resources to route traffic. In our Helm chart, we've made this configurable through the `values.yaml` file. When you enable `gatewayApi.gateway.enabled` and `gatewayApi.httpRoute.enabled`, Helm generates resources similar to the examples below.
#### Example Gateway Configuration

This YAML defines a `Gateway` that will be managed by Istio. It listens on port 80 for HTTP traffic and allows routes from any namespace to attach to it. This resource is typically managed by a cluster administrator.

```yaml
# This is an example of the Gateway resource generated by the Helm chart.
apiVersion: gateway.networking.k8s.io/v1beta1
kind: Gateway
metadata:
  name: app-k8s-gateway      # The name of our Gateway
  namespace: istio-system         # Deployed in the same namespace as the Istio controller
spec:
  gatewayClassName: istio         # Specifies that Istio should manage this Gateway
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: All                 # Allows HTTPRoutes from all namespaces to attach
```

#### Example HTTPRoute Configuration

This YAML defines an `HTTPRoute` that attaches to the `Gateway` we created above. It matches all incoming requests (`/`) and forwards them to our application service on port 3001. This resource is typically managed by an application developer.

```yaml
# This is an example of the HTTPRoute resource generated by the Helm chart.
apiVersion: gateway.networking.k8s.io/v1beta1
kind: HTTPRoute
metadata:
  name: app-http-route
  namespace: default           # Deployed in our application's namespace
spec:
  parentRefs:
  - name: app-k8s-gateway     # Attaches to our Gateway...
    namespace: istio-system       # ...which is in the istio-system namespace
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /                  # Matches all requests
    backendRefs:
    - name: prime-calculator               # Forwards traffic to our service...
      port: 3001                    # ...on port 3001
```

**Important Note on Finding the IP Address**:

When you create a `Gateway` resource that is managed by Istio, Istio will provision a new `Service` of type `LoadBalancer` specifically for that gateway. This is different from the default `istio-ingressgateway` service.

To find the correct external IP address for your application, you need to look for the service associated with your new gateway.

```sh
# List the services in the istio-system namespace
kubectl get svc -n istio-system
```

In the output, you will see a new service, likely named after your `Gateway` resource (e.g., `app-k8s-gateway-istio`). You must use the `EXTERNAL-IP` of this new service to access your application.
